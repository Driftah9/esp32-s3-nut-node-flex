# v0.44 Table-Driven Architecture

## Rationale

Previous versions (v0.40-0.43) accumulated hardcoded RID arrays and per-vendor decode functions directly in `ups_get_report.c`. This made it difficult to add new devices without modifying the core polling logic.

v0.44 introduces a **table-driven design** that mirrors NUT's subdriver pattern, allowing new devices to be added by updating the device database only.

## Core Components

### 1. `hid_get_report_info_t` (ups_device_db.h)

Structure describing one Feature report to poll:

```c
typedef struct hid_get_report_info {
    uint8_t     rid;         /* Report ID (e.g. 0x50 for ups.load) */
    const char  *nut_var;    /* NUT variable name (e.g. "ups.load") */
    unsigned    flags;       /* HU_FLAG_* poll mode hints */
    void        (*decode_fn)(uint8_t rid, const uint8_t *data, size_t len);
} hid_get_report_info_t;
```

### 2. `ups_device_entry_t.get_report_table` (ups_device_db.h)

Each device entry now points to an optional table of Feature report RIDs to poll:

```c
const hid_get_report_info_t s_apc_backups_get_report_table[] = {
    { .rid = 0x17, .nut_var = "input.voltage", .flags = 0, .decode_fn = decode_apc_backups_rid_0x17 },
    { .rid = 0x50, .nut_var = "ups.load",      .flags = 0, .decode_fn = decode_apc_backups_rid_0x50 },
    { .rid = 0,    .nut_var = NULL,             .flags = 0, .decode_fn = NULL },  /* terminator */
};
```

### 3. Vendor-Specific Decode Functions

Moved from `ups_get_report.c` to vendor files (`ups_db_apc.c`, etc.):

```c
static void decode_apc_backups_rid_0x50(uint8_t rid, const uint8_t *data, size_t len)
{
    if (len < 2u) return;
    uint8_t load = data[1];
    if (load <= 100u) {
        ups_state_update_t upd;
        memset(&upd, 0, sizeof(upd));
        upd.valid          = true;
        upd.ups_load_valid = true;
        upd.ups_load_pct   = load;
        ups_state_apply_update(&upd);
    }
}
```

## APC Implementation (v0.44-alpha)

### Back-UPS (PID 0x0002)

| RID  | NUT Variable  | Source | Format | Notes |
|------|---------------|--------|--------|-------|
| 0x17 | input.voltage | Feature | uint16 LE, 1V units | AC line voltage |
| 0x50 | ups.load      | Feature | uint8, 0-100% | PowerConverter.PercentLoad |

### Smart-UPS (PID 0x0003)

| RID  | NUT Variable       | Source | Format | Notes |
|------|-------------------|--------|--------|-------|
| 0x06 | ups.status        | Feature | byte[1]=charging, byte[2]=discharging | TODO: integrate flags |
| 0x0E | battery.voltage    | Feature | uint8, 10mV units | Battery voltage |

## Migration Path

### Immediate (v0.44-alpha)

### Next Phase (v0.44-beta)
- [ ] Refactor `ups_get_report.c` to generic table walker

  - Loop device_entry->get_report_table instead
- [ ] Build CyberPower Feature polling table from NUT cps-hid.c

- [ ] Refactor `nut_server.c` to loop `ups_var_store` instead of hardcoded variable list

- [ ] Release as stable v1.0

## Benefits

1. **Scalable**: Adding a new device requires only adding a new entry in the device database and its decode functions. No changes to core polling logic.
2. **Maintainable**: All vendor-specific logic is in vendor files, not mixed with generic code.
3. **NUT-aligned**: Mirrors the subdriver pattern, making port updates from upstream NUT easier.
4. **Testable**: Each device's table and decode functions can be unit-tested in isolation.

## Known Gaps (Future Work)

- Smart-UPS rid=0x06 charging flags not yet integrated into ups_status (needs ups_state_update_t extension)
- No Feature polling for CyberPower (0x0764) yet - all data comes from interrupt-IN
- No Feature polling for generic devices - only confirmed for APC/Eaton/PowerWalker

## References

- NUT cps-hid.c: CyberPower decode patterns (vendor RID 0x20-0x88)
- NUT mge-hid.c: Eaton complex multi-path patterns with 0xFFFF vendor page
